PART 1 — UNIVERSAL PAGINATION CHANGE ⚠️ Breaking
Every list endpoint switched from offset pagination to cursor (keyset) pagination. This is the most impactful change.

Response shape — before vs after
Before (every list):


{
  "success": true,
  "data": [ {...}, {...} ],
  "meta": {
    "total": 247,
    "limit": 50,
    "offset": 100,
    "has_next": true,
    "has_prev": true
  }
}
After (every list):


{
  "success": true,
  "data": {
    "items": [ {...}, {...} ],
    "meta": {
      "page_size": 20,
      "next_cursor": "MjAyNi0wNS0wMVQxMDowMDowMCswMDowMHwxMjM0NQ",
      "has_next": true
    }
  }
}
Field	Before	After
data	Array directly	Object with items + meta
data.items	Did not exist	Array of items
meta.total	Integer count	Removed
meta.limit	Items per page	Removed
meta.offset	Current offset	Removed
meta.has_prev	Boolean	Removed
meta.page / meta.page_size	Various	Removed
meta.next_cursor	Did not exist	Opaque string or null
meta.has_next	Boolean	Boolean (kept)
meta.page_size	Did not exist	Integer
How to paginate — code pattern

// FIRST PAGE — no cursor
const res = await api.get(`/endpoint?page_size=20`);
const { items, meta } = res.data.data;

// NEXT PAGE — pass cursor back
if (meta.has_next) {
  const next = await api.get(`/endpoint?page_size=20&cursor=${meta.next_cursor}`);
}

// LAST PAGE
// meta.has_next === false && meta.next_cursor === null

// INVALID CURSOR — server returns HTTP 400:
// { "success": false, "detail": "Invalid pagination cursor: '...'" }
// Handle by fetching page 1 again (drop the cursor)
Back button / "Page N of M"
Cursor pagination is forward-only. Options:

Use a "Load More" button (no back needed) — recommended
Keep a cursor stack client-side: push on Next, pop on Back
total is not returned — remove any "X of Y results" counter, or call a separate count endpoint

PART 3 — NEW STRIPE ENDPOINTS
3.1 Start checkout (subscribe to a plan)

POST /api/v1/billing/stripe/{tenant_public_id}/checkout
Auth required. Creates a Stripe Checkout Session.

Request:


{
  "plan_public_id": "abc123",
  "billing_interval": "monthly",
  "success_url": "https://yourapp.com/billing/success?session_id={CHECKOUT_SESSION_ID}",
  "cancel_url": "https://yourapp.com/billing/cancelled",
  "promotion_code": null
}
billing_interval: "monthly" or "annual". success_url / cancel_url / promotion_code are optional.

Response:


{
  "success": true,
  "data": {
    "session_id": "cs_test_abc123...",
    "url": "https://checkout.stripe.com/pay/cs_test_abc123..."
  }
}
Frontend flow:


const res = await api.post(`/billing/stripe/${tenantId}/checkout`, {
  plan_public_id: plan.public_id,
  billing_interval: "monthly"
});
window.location.href = res.data.data.url;  // redirect to Stripe
3.2 Billing self-service portal

POST /api/v1/billing/stripe/{tenant_public_id}/portal
Auth required. Opens Stripe's hosted portal — Stripe handles card management, invoices, plan changes, cancellation. No UI to build on your side.

Request:


{ "return_url": "https://yourapp.com/billing" }
Response:


{
  "success": true,
  "data": { "url": "https://billing.stripe.com/session/..." }
}
Frontend flow:


const res = await api.post(`/billing/stripe/${tenantId}/portal`, {});
window.location.href = res.data.data.url;
Use this for: "Manage Card", "View Invoices", "Change Plan", "Cancel Subscription" buttons.

3.3 Webhook — no frontend work needed
POST /api/v1/webhooks/stripe — Stripe calls this server-to-server automatically after payment. Nothing for frontend to do.

const requestId = response.headers['x-request-id'];

PART 6 — CHATBOT EMBED — DOMAIN ENFORCEMENT

GET /api/v1/chatbot/embed/{iframe_token}
The embed endpoint now validates the Origin / Referer header against the chatbot's allowed_domains list.

If the widget is embedded on a domain not in the allowed list → HTTP 403.

If your widget starts returning 403: add your frontend domain to the chatbot's allowed_domains config via the admin API. Wildcards are supported: *.yourdomain.com.

PART 7 — CHATBOT FALLBACK MESSAGE (behaviour change, not API)
When OpenAI is unreachable (circuit breaker open), the chat endpoint now returns:

"The AI service is temporarily unavailable. Please try again in a moment."

Previously this would hang or return a generic error. No API shape change — the chatbot message field is still a string. No frontend code change needed, but this is good to know for customer support.
